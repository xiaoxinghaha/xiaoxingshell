- 现象 1：
  窗口最小化后再恢复，终端偶发只剩 5 行显示。
- 原因 1：
  最小化或暂时不可见时，界面层会短暂测到一个退化尺寸，之前这组无效尺寸会继续传到 PTY 和终端解析器里。
- 修复 1：
  界面层只会在终端尺寸真实有效时才发送 resize；Rust 侧也补了一层保护，忽略明显属于最小化瞬间的临时 `10x5` 尺寸。
- 现象 2：
  在前一轮为恢复显示而加入同步刷新后，点击连接 SSH 时存在触发终端刷新重入的风险，导致应用退出。
- 修复 2：
  终端 resize 后的界面刷新改为走已有的异步刷新队列，不再在 resize 回调里同步重建显示。
- 结果：
  终端从最小化恢复后会按原来的可见区域重新显示，不会再被错误压成 5 行；连接 SSH 时的稳定性也更好。

涉及文件：

#### 2. 修复 SSH 首次连上后注入的提示符 Hook 干扰终端体验的问题

- 现象 1：
  刚连上服务器时，终端为了安装 cwd / 命令捕获 Hook，会注入一段较长的 shell 命令；在某些服务器环境下，这段命令会出现在上方向键历史里。
- 修复 1：
  调整了注入脚本中的历史清理逻辑，尽量在 bash / zsh 场景下把这条内部命令从 shell 历史里移除，避免用户按上键时看到这段初始化命令。
- 现象 2：
  注入阶段如果用户很快输入内容，偶发会出现终端“吃掉一部分字符”或回显混乱。
- 修复 2：
  在初始化回显抑制阶段增加输入缓冲，等 Hook 安装完成后再按顺序回放用户输入，避免早输入的字符被初始化输出一起吞掉。
- 结果：
  SSH 首次连接后的终端初始化更稳，历史记录和输入体验都更接近正常 shell 行为。

涉及文件：

- `src/ssh.rs`

#### 3. 修复 SFTP“跟随 cd 更新”误触发的问题

- 现象：
  开启“SFTP 跟随 cd 更新”后，哪怕只是空回车、执行 `ls` / `pwd` 这类普通命令，SFTP 面板也可能被拉回终端当前目录。
- 原因：
  之前的跟随逻辑主要依赖提示符阶段上报的 cwd；而 shell 每次显示新 prompt 时都会重复上报 cwd，导致“并没有执行 cd”也可能触发同步。
- 修复：
  现在只有终端里真实执行了 `cd` 命令，后续 cwd 变化才会驱动 SFTP 跟随。手动在 SFTP 面板中浏览目录后，空回车或普通命令也不会再强行把面板拉回去。
- 结果：
  “跟随 cd”只在真正需要时工作，不会再因为普通回车或其他命令把 SFTP 面板错误跳转。

涉及文件：

- `src/app.rs`
- `src/ssh.rs`

#### 4. 修复第一次执行 `cd /xxx` 时 SFTP 不立即跟随的问题

- 现象：
  在上一版“只对真实 `cd` 跟随”修复后，部分服务器上第一次输入 `cd /xxx` 时，SFTP 不会立刻更新；需要第二次执行 `cd` 才能跟随成功。
- 原因：
  不同 shell / 服务器环境下，`命令捕获` 与 `cwd 上报` 的事件顺序并不完全一致；有些情况下 cwd 先到、命令后到，导致第一次 `cd` 被错过。
- 修复：
  调整事件判定逻辑，兼容“先收到命令再收到 cwd”和“先收到 cwd 再收到命令”两种顺序。
- 结果：
  现在第一次执行 `cd /xxx` 时，SFTP 就会立即跟随到对应目录。

涉及文件：

- `src/app.rs`

#### 6. 修复下载取消后的回归问题
